## 1. 项目背景

### 1.1 行业现状

AI行业每天产生大量新闻、工具、开源项目、创业案例和招聘信息。普通用户每天需要浏览 OpenAI Blog、Anthropic Blog、Google AI Blog、DeepSeek、GitHub Trending、Hacker News、Product Hunt 等7个以上信息源，获取有效信息需要花费大量时间。

### 1.2 核心矛盾

| 矛盾 | 说明 |
|------|------|
| 信息过载 vs 有效信息稀缺 | 每天500+篇AI相关内容，但真正重要的通常只有5-10条 |
| 英文为主 vs 中文用户需求 | 90%的AI一手信息为英文，中文用户存在语言门槛和信息延迟 |
| 信息分散 vs 整合需求 | 信息分布在7+个平台，缺乏统一的聚合视图 |
| 信息获取 vs 价值判断 | 用户不仅需要"看到"信息，更需要"判断"哪些值得投入时间 |

### 1.3 产品定位

AI机会雷达是一款面向AI从业者、独立开发者和技术创业者的信息聚合与分析平台。系统自动采集全球主流AI信息源，利用大语言模型进行深度分析，每日生成高质量的AI机会日报，帮助用户在5分钟内掌握AI领域的关键动态与商业机会。

### 1.4 产品使命

减少AI从业者的信息筛选成本，让每个人都能快速发现AI浪潮中的学习机会、商业机会和职业机会。

## 3. 用户痛点

### 3.1 痛点矩阵

| 痛点 | 频率 | 强度 | 现有方案 | 我们的方案 |
|------|------|------|---------|----------|
| 每天需要浏览6+网站 | ⭐⭐⭐ | ⭐⭐⭐ | Feedly / 手动收藏夹 | 自动聚合一个不落 |
| 英文资讯阅读耗时 | ⭐⭐⭐ | ⭐⭐⭐ | Google翻译 / 硬读 | AI自动摘要+中文 |
| 难判断信息价值 | ⭐⭐⭐ | ⭐⭐⭐ | 依赖个人经验 | AI多维评分体系 |
| 错过重要开源项目 | ⭐⭐ | ⭐⭐⭐ | GitHub Watch / 社交网络 | 自动追踪+分析推荐 |
| 不了解岗位技能趋势 | ⭐⭐ | ⭐⭐ | LinkedIn手动搜 | 自动统计+学习建议 |
| 发现商业机会靠偶然 | ⭐ | ⭐⭐⭐ | 无系统方法 | AI扫描+商业模式分析 |

### 3.2 关键洞察

用户真正关心的不是"今天有什么新闻"，而是：

1. **哪些内容值得学习** — 不要把时间花在过时的技术上
2. **哪些内容影响就业** — 技能市场在变化，需要及时调整方向
3. **哪些内容存在商业机会** — 别人已经在赚钱的方向
4. **哪些项目值得复刻** — 站在开源巨人的肩膀上
5. **哪些技术值得投入时间** — 区分炒作和真正的趋势

## 4. 产品目标

### 4.1 核心目标

每天8点自动生成并推送高质量AI机会日报，用户阅读时间控制在5分钟以内。

### 4.2 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|-------|---------|
| 日报阅读率 | >70% | 推送打开数 / 推送总数 |
| 阅读完成率 | >60% | 阅读到底部 / 打开用户数 |
| 信息覆盖率 | 100% | 覆盖全部7个核心源 |
| 分析准确率 | >4.0/5.0 | 人工抽样评估AI分析质量 |
| 系统可用性 | >99.5% | 月正常运行时间 |
| 推送成功率 | >99% | 成功推送 / 尝试推送总数 |

### 5.2 日报结构

日报包含以下6个板块，总阅读时间控制在5分钟以内：

| 板块 | 内容 | 数量 | 阅读时间 |
|------|------|------|---------|
| 今日重点新闻 | 最重要的AI新闻，含摘要和影响分析 | 3-5条 | 1分钟 |
| AI工具推荐 | 新发布的AI工具和产品 | 2-3个 | 45秒 |
| GitHub热门项目 | 值得关注的开源项目 | 3-5个 | 1分钟 |
| 创业机会分析 | 可商业化的AI产品方向 | 2-3个 | 1分钟 |
| AI岗位趋势 | 技能变化和市场需求 | — | 45秒 |
| 今日行动建议 | 具体可执行的操作建议 | 2-3条 | 30秒 |

### 5.3 数据采集源

| 来源 | 类型 | 采集频率 |
|------|------|---------|
| OpenAI Blog | Blog | 每2小时 |
| Anthropic Blog | Blog | 每2小时 |
| Google AI Blog | Blog | 每2小时 |
| DeepSeek Blog | Blog | 每2小时 |
| GitHub Trending | 社区 | 每6小时 |
| Hacker News | 社区 | 每1小时 |
| Product Hunt | 社区 | 每6小时 |

### 5.4 AI评分体系

每条资讯从4个维度进行1-10分评分：

| 评分 | 1-3分 | 4-6分 | 7-8分 | 9-10分 |
|------|-------|-------|-------|--------|
| 重要程度 | 无关信息 | 普通更新 | 重要发布 | 里程碑事件 |
| 学习价值 | 无需学习 | 了解即可 | 值得投入 | 必须掌握 |
| 商业价值 | 无商业价值 | 间接价值 | 直接商机 | 重大商机 |
| 创业价值 | 不适合创业 | 可参考 | 值得复制 | 立即行动 |

同时自动判定关注等级：urgent 🔴 / high 🟠 / normal ⚪ / low ⬜

### 5.5 推送方式

| 渠道 | 优先级 | 说明 |
|------|--------|------|
| 企业微信机器人 | P0 | 每日08:00推送Markdown格式日报 |
| 邮件推送 | P1 | 每日08:00推送HTML格式日报 |
| 飞书/钉钉 | 后续 | 扩展推送渠道 |

## 7. 用户流程

### 7.1 场景一：每日日报阅读

```
08:00 收到推送
  → 点击链接打开日报Web页面
  → 浏览今日重点新闻（30秒）
  → 查看AI工具推荐（45秒）
  → 浏览GitHub热门项目（60秒）
  → 查看创业机会分析（60秒）
  → 浏览AI岗位趋势（45秒）
  → 查看今日行动建议（30秒）
  → 完成阅读，总计约5分钟
```

### 7.2 场景二：PC浏览器查看历史日报

```
打开浏览器 → 访问系统URL → 看到今日日报
  → 点击"历史日报" → 选择目标日期
  → 查看当日完整日报
  → 可翻页浏览更早日期
```

### 7.3 场景三：手机查看日报

```
收到推送链接 → 点击 → 适配移动端布局
  → 上下滑动浏览各板块
  → 点击卡片展开详
  → 支持暗黑模式
```

### 7.4 场景四：查看近7天热点

```
Web后台 → 点击"热点趋势"
  → 查看近7天热点分布
  → 点击热点词查看相关资讯
  → 了解趋势变化
```

### 7.5 异常流程

| 场景 | 处理方式 |
|------|---------|
| 推送未收到 | 用户可手动访问Web后台查看最新日报 |
| 日报生成失败 | 自动重试一次，15分钟未成功则跳过当日 |
| 部分数据为空 | 保留板块结构，标注"今日暂无数据" |
| 采集全部失败 | 记录日志，跳过日报生成，次日正常采集 |

## 8. 非功能需求

### 8.1 系统要求

| 维度 | 要求 |
|------|------|
| 稳定性 | 单模块故障不影响其他模块，异常自动重试 |
| 可用性 | >99.5%（月），定时任务失败自动恢复 |
| 性能 | 日报页面加载<2秒，API响应<500ms |
| 安全性 | API Key加密存储，SQL注入防护，XSS防护 |
| 可维护性 | 模块化设计，完整日志，30天日志保留 |

### 8.2 数据要求

| 维度 | 要求 |
|------|------|
| 完整性 | 写入失败自动回滚，数据不丢失 |
| 一致性 | 模块间通过数据库状态解耦 |
| 备份 | 每日自动备份，保留7天 |
| 清理 | 日志保留30天自动清理 |

## 9. 产品路线图

### V1.0 — MVP（14天）

核心链路跑通：自动采集 → AI分析 → 日报生成 → 企业微信/邮件推送。单用户Web后台可查看日报和管理配置。

### V1.1 — 增强（发布后1-2周）

- Redis缓存提升日报加载速度
- 自定义采集源配置
- 飞书/钉钉推送渠道
- 日报PDF导出
- 资讯全文搜索

### V2.0 — 多用户（发布后1个月）

- 用户注册登录（JWT/邮箱）
- 多用户订阅与偏好设置
- PostgreSQL数据库迁移
- 个人收藏/标记功能
- 日报点赞/反馈

### V2.5 — 智能（发布后2个月）

- AI对话式查询
- 个性化内容推荐
- 热点趋势预测
- 竞品自动追踪
- 周报/月报自动生成

### V3.0 — 商业化（发布后3个月）

- SaaS多租户架构
- 付费订阅体系
- 团队协作与分享
- API开放平台
- 数据报表导出
